历史与分层模型

tcp/ip由美军的ARPANET发展而来，1983年成为其上的标准协议，因此1983年也被称为互联网诞生元年。tcp/ip分为：

  • 应用层
  • 传输层
  • 网络互连层
  • 网络访问层

应用层

本质上规定了应用程序之间如何传递信息，应用层诸如http，就规定了报文的一些格式；其他协议诸如DNS,SMTP,POP3,NTP(时钟同步协议),NFS也类似

传输层

TCP就被称为传输控制协议，规定了诸如端口号的功能，使数据能够正确传到相应的应用程序。为下面的两层提供数据包的重传，流量控制，拥塞控制等。

网络互连层

ip协议提供主机到主机的通信，tcp/udp都是采用ip作为网络层协议，其是无连接的——即不管数据是否被发送到目标主机，也不管数据包是否传输错误。

网络访问层

提供主机到物理网络需要的硬件和相关协议，诸如蓝牙，wifi就工作在这一层

  • 收到 IP 数据包解析以后，它怎么知道这个分组应该投递到上层的哪一个协议（UDP 还是 TCP）？
  • 正如上文所述，IP 头中有个字段 protocol 可用来表示上层用的哪个协议，ip协议为上层的协议提供了基础

TCP概述

面向连接

什么叫做面向连接？即在正式发送数据前通过握手建立一个逻辑连接，结束时通过有序的挥手断开连接。无连接的ip则不需要。

可靠

tcp之所以是可靠的，原因在于：

  • 对每一个包提供校验和
  • 每一个包都有序列号
  • 超时重传
  • 流量控制、拥塞控制
    校验和（checksum）：每个 TCP 包首部中都有两字节用来表示校验和，如果收到一个校验和有差错的报文，TCP 不会发送任何确认直接丢弃它，等待发送端重传。而不是试着纠错或者立即要求重新传送。

超时重传 TCP 发送数据后会启动一个定时器，等待对端确认收到这个数据包。如果在指定的时间内没有收到 ACK 确认，就会重传数据包，然后等待更长时间，如果还没有收到就再重传，在多次重传仍然失败以后，TCP 会放弃这个包。

面向字节流

所谓字节流，指数据像水一般流动，比如有一个1300字节的数据，tcp可能把他以500、800发出，也有可能以1000、300发出，也有可能以1300直接发出。发出的大小自然取决于路径最大传输单元 MTU、发送窗口大小、拥塞窗口大小等。

全双工

在 TCP 中发送端和接收端可以是客户端/服务端，也可以是服务器/客户端，通信的双方在任意时刻既可以是接收数据也可以是发送数据，每个方向的数据流都独立管理序列号、滑动窗口大小、MSS 等信息。

tcp自连接

TCP 的自连接是一个比较有意思的现象，甚至很多人认为是 Linux 内核的 bug。我们先来看看 TCP 的自连接是什么。

作者实验后发现，

源 IP、源端口是 127.0.0.1:50000，目标 ip、目标端口也是 127.0.0.1:50000

抓包，

根据作者的描述，似乎一个服务不断尝试重连后就可能出现自连接的情况，但是为啥啊？先挖个坑。。。

解决自连接问题有两种办法：

  • 让服务监听端口与客户端随机分配的端口不可能相同即可
  • 主动关掉自连接

四次挥手

  • 客户端调用close方法，发送一个FIN报文给客户端，之后客户端将只能接收数据，即进入FIN-WAIT-1状态。FIN消耗一个序列号
  • 服务端收到后回复ACK报文，服务端进入CLOSE_WAIT，客户端收到ACK报文后进入FIN-WAIT-2状态
  • 服务端此时发送一个FIN报文给客户端，进入LAST-ACK状态，也消耗一个序列号
  • 客户端回复ACK报文，进入TIME_WAIT状态，等待2个MSL后进入CLOSED状态，服务端收到ACK后进入CLOSED状态

TCP11种状态变迁以及模拟重现

tcp的复杂在于其状态繁多，但正是其11种状态，保证了该协议的安全可靠。

closed

是一个假想的状态，无法通过netstat,lsof等工具看到。仅是TCP连接还未开始建立以及连接彻底释放时的状态

LISTEN

一端通过bind.listen调用短句进入该状态，等待另一端发送SYN报文三次建立连接

SYN-SENT

一端发送SYN报文后等待ACK的过程即为SYN-SENT状态

SYN-RCVD

一端回复SYN+ACK后等待对端ACK的状态

ESTABLISHED

上面的SYN-SENT和SYN-RCVD收到ACK后进入该状态。即连接建立成功

FIN-WAIT-1

主动关闭的一方发送FIN包后等待对端ACK时进入该状态

FIN-WAIT-2

FIN-WAIT-2收到ACK后进入该状态

CLOSE-WAIT

上述中收到FIN包的一端，在收到时进入CLOSE-WAIT状态

TIME-WAIT

发起关闭的一方收到了对方的FIN包，发送ACK给对端，开启定时器，定时器到期进入CLOSED，连接释放。

LAST-ACK

被动关闭的一方发送FIN包给对端等待ACK的状态。收到ACK后进入CLOSED状态

CLOSING

仅在主动关闭一端发送FIN包后还未收到ACK时，收到了对端的FIN包

全连接队列，半连接队列，backlog

基本概念

先看listen函数:

调用listen函数时，TCP的状态从CLOSE变为LISTEN，内核创建两个队列：

半连接队列，incomplete connection queue,即SYN队列

全连接队列，completed connection queue,即accept队列

所谓 半连接队列，指的是客户端发送SYN到服务端后，服务端回复自己的ACK和SYN，服务端的TCP从listen到SYN_RCVD，这个连接信息放入服务端的半连接队列，存储所谓的inbound SYN packets

当服务端收到了客户端的ACK后，就把这个信息加入全连接队列

「全连接队列」包含了服务端所有完成了三次握手，但是还未被应用调用 accept 取走的连接队列。

全连接队列又被称为ACcept队列，如果队列为空，accept通常会阻塞。

SO_REUSEADDR

我们假设这样一个情景：客户端与服务器建立连接，突然服务器挂了。这个时候连接会断开，等待两个MSL后最终释放连接。假设最终释放前服务器重启完毕，之前建立连接的那个程序也启动，这时就会出现 address already in use

由于TCP由唯一的四元组确定，而大多数OS会阻止新的监听套接字绑定到已经建立连接的端口上，则bind会调用失败，出现错误。

而SO_REUSEADDR可以解除这个限制，也允许端口重复

epoll惊群

linux为了表示打开文件，其内核会维护三种数据结构：

  1. 为每一个进程维护了一个打开文件的描述符表，比如fd为0的stdin就属于此
  2. 内核为所有打开文件维护了一个系统级的打开文件表，这个表存储了当前文件的偏移量等信息。而父子进程的fd可以指向同一个打开文件表
  3. 文件系统的inode表

目前linux内核没有提供epoll惊群的解决办法

相比于之前的SO_REUSEADDR，SO_REUSEPORT的实现要复杂得多。

内核会为处于listen状态的socket分配一个hash,监听的端口号会经过hash打散到这些hash桶中，采用拉链法（类似于邻接表）解决冲突。在收到syn报文后，内核根据目标端口号算出链表，遍历此链表寻找最匹配、得分最高的socket。如果多个socket得分最高，则会随机选择一个。

SO_REUSE_PORT带来的好处自然是内核级的负载均衡，另一个好处是负载均衡，什么叫负载均衡？

  1. 新启动一个新版本 v2 ，监听同一个端口，与 v1 旧版本一起处理请求。
  2. 发送信号给 v1 版本的进程，让它不再接受新的请求
  3. 等待一段时间，等 v1 版本的用户请求都已经处理完毕时，v1 版本的进程退出，留下 v2 版本继续服务

time_wait

只有主动断开连接的哪一方会进入time_wait

在tcp协议中,MSL为报文在网络中的最大生存时间，与ip报文中的ttl字段有密切的关系。在ip协议中,ttl是用来表示一个报文最大可以经过的路由数，

我们假设最大跳数的报文在网络中的存活时间不可能超过msl秒，那么tcp中的time_wait将持续2*msl秒。

time_wait存在的原因

  • 等待老的重复报文在网络中过期失效。这样可以避免同样的tcp四元组在新连接收到旧连接的数据
  • 确保tcp的全双工可靠。由于主动关闭的一方发送ack后需要被动方发送fin，如果主动关闭方发送的Ack丢失，而主动关闭方直接closed,则无法重传丢失的ack，被动关闭方不能及时可靠释放。

time_wait产生的问题

大量的time_wait状态会造成两个问题，

  1. 连接表无法复用
    显然，在time_wait状态，相同的tcp四元组无法被复用
  2. socket结构体内存占用

解决问题

开启时间戳后，可以用tcp_tw_resue,开启后可以重用出于浪费状态的连接，有两种情况：

  1. 旧连接ack包未丢失，此时由时间戳可以判断是否应该丢弃数据
  2. 旧的ack包丢失，被动方出于last-ack状态，持续重传fin+ack；主动方收到被动方的fin后回一个rst，被动方关闭。主动方由于未收到syn的ack，会继续重传，此后正常进行。

tcp_tw_recycle，系统缓存 每台主机（ip）的最新时间戳，通过时间戳比较确定该丢弃的数据包。他会带来一个问题，在客户端和服务端一对一的情况下还好，假设客户端经过了nat或负载均衡，就会造成服务端误以为在和一台主机通信，仅会接受最新时间戳的连接。

NAT

所谓nat,就是为了缓解ipv4地址耗尽的方案，nat设备负责维护局域网私有ip和端口到外网ip和端口的映射规则，其优点在于通过一个公网ip让许多机器连接网络，让其隐藏真实ip，缺点在于一些协议无法通过NAT。

RST

rst用来表示异常的关闭连接。比如客户端发送syn包到指定端口，服务端未开启这个端口，则会回复rst。或者在建立连接后，一方突然断电，另一方在发送数据后就会收到断电方回复的rst。总之就是表示一个异常的关闭。如果传输数据时，一方的rst包丢失，另一方未收到，则会一直传输直到收到rst包或者达到一定次数后放弃

brocken_pipe 和connection reset by peer

所谓brocken pipe，就是

即向rst中执行写操作的错误。connection reset by peer则是执行读操作

重传

我们假设一个这样的情景，服务端想客户端发送5000个字节的数据包，由于MSS的限制而每次传输1000个字节，即1：1001，1001：2001，2001：3001，3001：4001，4001：5001.假设第二个数据包丢失，在服务端重传第二个数据包之前第三，四，五个数据包已经到达，但由于第二个数据包未收到，因此服务端无法收到客户端的第二个数据包的ack，客户端也无法回复第三，四，五数据包的ack。在第二个数据包到达后客户端才会回复ack5。在网络中，要等几百毫秒才能重传，这显然是过慢的。于是有了快速重传机制，SACK。

快速重传与SACK

快速重传是什么？——在上面的例子中，客户端收到3，4，5包后回复的还是ack1,服务端收到重复的ack1后可以确定2包丢失而不必等待网络重传，这就是快速重传

SACK是什么？——在之前的例子中，客户端只能回复ack1，那么SACK就是在客户端收到3的时候在ack中回复自己收到的包，[1:1001]、[2001:3001] 区间的包我也收到了,之后的4.5包也一样。这样服务端清楚的知道到底哪一个包需要重传了

发送端收到三次重复ack后自动触发快速重传

重传的间隔确定

间隔多久才能重传显然要根据网络情况进行一个动态的调整，超时重传的时间叫做RTO，计算的方法有两种：

  1. 经典方法，计算平均值
  2. 标准方法

除此之外，还要兼顾一下在重传的情况下如何确定RTT。下面分别摘抄一下

经典算法

适用于RTT波动较小的情况，其实就是引入了一个计算平均值的方法。

在经典算法中，引入了平滑往返时间SRTT的概念，即每测量一次RTT，就更新一次SRTT,$SRTT = ( \alpha \times SRTT ) + ((1-\alpha) \times RTT)$作者提了一个很魔鬼有趣的比喻，假设平滑因子$\alpha=0.8$,那么SRTT就相当于一个低通滤波器。

显然，$\alpha$趋近于1时对短暂的实验变化不敏感，趋近于0时则对时延变化十分敏感而在经典算法的基础上，计算RTO的公式则是:$RTO= min(ubound,max(lbound,\beta \times SRTT))$，其中$\beta$为加权化因子，推荐值为1.32.0，![](https://p1-jj.byteimg.com/tos-cn-i-t2oaga2asx/gold-user-assets/2019/4/5/169ee07dda466279tplv-t2oaga2asx-watermark.awebp)

经典算法在稳定的RTT环境下表现良好，在网络不稳定的情况下则效果极差

标准方法，jacobson/karels算法

经典算法的问题在于$\alpha$对效果影响很大，而即便能够实时更新$\alpha$，其线性表达式仍不够好。jacobson/karels算法如下：

$SRTT=(1-\alpha) \times SRTT + \alpha \times RTT$

$RTTVAR=(1- \beta)\times RTTVAR+\beta \times (| RTT-SRTT |)$

$RTO=\mu \times SRTT + \partial \times RTTVar$

第一个公式就是经典的计算公式，第二个公式就是计算RTTVar,已平滑的RTT平均偏差估计器，第三个公式则计算最终的RTO值。上述公式中，$\alpha$建议0.125，$\beta$建议0.25，$\mu$建议1，$\partial$建议4

重传与karn/partridge算法

一个最基本的问题是，如何在重传情况下计算RTT.

上图中指明了重传二义性，

  • 如果用第一次发送数据的时间和收到 ACK 的时间来算 RTT，就会出现图 1 和图 2 中的问题，RTT 时间明显是大于实际值
  • 如果用第二次发送数据的时间和收到 ACK 的时间差值来算 RTT，就会出现图 3 中的问题，RTT 时间明显小于实际值

karn/partridge算法是怎么解决重传二义性的？不解决它。

即这次重传的RTT不会被用来更新SRTT以及RTO.当然，如果网络出现大规模抖动，忽略是不行的，这是karn采用了将RTO翻倍的暴力方法，产生一个指数级退避，粗暴但是有效

流量控制——滑动窗口

众所周知，tcp协议要求把发送的数据放入发送缓冲区，接收的数据放入接受缓冲区。流量控制要做的，就是管理发送端的发送速率，防止在接收缓冲区已满的情况下发送端还会再发送数据。tcp要求再收到数据包后回复的ack包里携带自己接收窗口的大小。

tcp把数据包的状态分为四种:

发送窗口表示发送端被允许发送的最大数据包大小，即在某一个时刻能拥有的最大未确认的数据包大小，为2和3的大小总和

可用窗口是发送端还能发送的最大数据包大小，即3的大小

当可用窗口的6个字节发送出去后，可用窗口减小到0，除非收到对端的ack，否则发送端不能发送数据

TCP Zero Window

假设一个高负载的网络状况，接受方没有快速处理缓冲队列中的数据，那么他的接收端窗口将变小，发送端的滑动窗口也将变小。当发送端的滑动窗口变为0时，由于网络高负载，发送端不知道接收端已经可以处理更多的数据包，那就只能一直等待接收端发送新的ack。这不行。

tcp设计了零窗口探测机制用于探测接收端的接收窗口。零窗口探测包就是一个0字节的ack，seq为当前seq最大值减1，发送端将一直发送这个包直到得到回应。

攻击

与之前介绍的 Syn Flood 攻击类似，上面的零窗口探测也会成为攻击的对象。试想一下，一个客户端利用服务器上现有的大文件，向服务器发起下载文件的请求，在接收少量几个字节以后把自己的 window 设置为 0，不再接收文件，服务端就会开始漫长的十几分钟时间的零窗口探测，如果有大量的客户端对服务端执行这种攻击操作，那么服务端资源很快就被消耗殆尽。

拥塞控制

拥塞控制主要考虑整个网络的通信状况，涉及到慢启动，拥塞避免，快速重传，快速恢复。tcp的每一条连接都有两个核心状态：

  • 拥塞窗口cwnd
  • 慢启动阈值ssthresh

拥塞窗口cwnd

指在收到对端ack之前自己还能传输的最大MSS段数

真正的发送窗口大小=min(接收端窗口，发送端拥塞窗口)

一、慢启动

网络拥堵时，会造成丢包严重，如果这个时候发送数据因丢包加多，会造成更大的网络延迟，基于整个网络考虑，1988年提出了慢启动算法。

这个算法的过程如下：

  • 第一步，三次握手以后，双方通过 ACK 告诉了对方自己的接收窗口（rwnd）的大小，之后就可以互相发数据了
  • 第二步，通信双方各自初始化自己的「拥塞窗口」（Congestion Window，cwnd）大小。
  • 第三步，cwnd 初始值较小时，每收到一个 ACK，cwnd + 1，每经过一个 RTT，cwnd 变为之前的两倍。

慢启动拥塞窗口有一个阈值，ssthresh,当cwnd<ssthresh时慢启，cwnd>ssthresh时拥塞窗口线性增长，这个阶段叫做拥塞避免，每经过一个RTT才增加拥塞窗口的值

二、快速恢复

之前介绍过快速重传，当收到三次重复ack时，进入快速恢复阶段，表现为网络轻度拥塞，此时ssthresh=cwnd/2,cwnd=ssthresh,拥塞窗口线性增加
